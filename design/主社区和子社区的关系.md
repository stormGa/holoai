# 主社区和子社区的关系

## 这是一个非常好的产品和增长策略问题。在分层架构中，确保用户流量分配得当，才能保证主社区的**经济安全**和子社区的**验证效率**。
我们的策略是：**将主社区定位为“高风险、高回报的治理层”，并将流量通过“智能引导”分发到最匹配的子社区。**

## 策略一：主社区的独特价值与激励

主社区必须拥有子社区无法替代的价值主张，才能吸引最顶尖的验证者。

| 区域   | 主社区的独特价值                                                            | 激励机制                                                                     |
| ---- | ------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| 内容风险 | 处理 \text{Cat} \text{I} 和 \text{Cat} \text{II} 的高风险内容（例如：全球政治、宏观金融）。 | 最高的奖励系数 (\text{R}_{\text{Risk}} = \mathbf{1.5x})。高风险内容吸引高声誉的验证者，保持主网的活力。 |
| 治理权限 | 是 \text{DAO} 提案和守护者投票的场所。                                           | 治理权是最高声望和权力的体现。只有参与主社区治理，才能获得守护者身份。                                      |
| 资金沉淀 | 社区领袖的 \mathbf{20,000} \text{TCC} 质押锁定在主社区的合约中。                      | 质押量巨大，主社区的经济安全性和权威性最高。                                                   |
| 内容发现 | 充当所有热门话题和通用新闻的总览中心 (\text{General} \text{Feed})。                    | 确保用户不会错过任何重大公共事件，形成每日必看的习惯。                                              |

## 策略二：智能引导与流量分发

我们利用 $$\text{API}$$ 逻辑和前端设计，引导用户找到最适合他们的社区，解决“如何让人优先用这个社区”的问题。

### 1. 强制性内容关联 (Content Association)

我们利用已设计的**“混合关联排序”**机制进行智能分流：

- **默认目标：** 当用户打开 $$\text{GET}$$ /feed/general 时，系统应**优先**展示与用户专业领域（$$\text{TopicBoost}$$）相匹配的**主社区** $$\text{PENDING}$$ 内容。
- **引导逻辑：** **除非**用户的专业领域与某个子社区的**核心标签**完全匹配，否则系统将主社区内容置于子社区内容之上，确保主社区的验证队列优先得到处理。

### 2. 用户兴趣标签与社区目录

- **Onboarding Flow：** 在用户注册时，通过问卷或兴趣标签（例如：$$\text{FINANCE}$$, $$\text{GAMING}$$, $$\text{SPORTS}$$），将用户的兴趣写入 `users` 表。
- **社区目录接口 (**$$\text{GET}$$** /communities)：** 建立一个公开的社区列表接口，允许用户按标签、活跃度或质押量 ($$\text{20,000}$$ $$\text{TCC}$$) 查找子社区。
   - **排序：** 优先展示与用户兴趣标签最匹配的子社区。
   - **视觉提示：** 在前端为用户未加入但兴趣匹配的子社区提供**“推荐”**或**“专业对口”**的视觉提示。

### 3. 用户体验 (UX) 优化

- **通用 Feed：** 默认展示主社区内容。
- **切换 Feed：** 提供清晰的**过滤器**和**侧边栏**，允许用户一键切换到他们已加入或被推荐的子社区 $$\text{Feed}$$。

---

**总结：** 主社区通过提供最高的经济风险和最大的治理影响力来吸引最精英的用户，而通过智能排序和清晰的目录来将普通用户导流到最相关的子社区。
